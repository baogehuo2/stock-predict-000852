中证1000短期趋势预测系统技术方案
一、项目目标

开发一个面向中证1000指数的短期趋势预测系统，预测未来：

3个交易日
5个交易日
7个交易日

输出内容包括：

1. 涨跌方向：上涨 / 下跌 / 震荡
2. 预测涨跌幅
3. 置信度
4. 技术面原因
5. 舆情热度原因
6. 大事件/预期差原因
7. 历史相似事件参考
8. 风险提示

本项目不使用 Tushare，优先使用免费数据源。

二、核心业务逻辑

系统不是简单做新闻情绪分析，而是判断：

技术趋势
+ 股民情绪热度
+ 新闻事件扩散
+ 预期是否已经交易
+ 官方消息是否存在利好兑现风险
+ 历史类似事件后续走势

核心判断框架：

预期形成 → 资金提前交易 → 股民讨论升温 → 新闻扩散 → 官方确认 → 利好兑现/继续发酵

中证1000属于小盘风格指数，短期受风险偏好、情绪资金和政策预期影响较大，因此需要同时结合：

技术指标
股吧热度
新闻事件
商品/股指期货情绪
历史相似事件

七、模块开发顺序
阶段1：行情数据与技术指标

目标：

先跑通行情采集、技术指标、标签构造、基础回测。

开发任务：

1. 创建MySQL数据库和表
2. 编写AKShare指数采集脚本
3. 采集中证1000、沪深300、中证500、创业板指、科创50
4. 写入market_index_daily
5. 计算技术指标
6. 写入market_feature_daily
7. 构造future_ret_3d/5d/7d和label
8. 训练LightGBM基础模型
9. 输出基础回测结果

验收标准：

1. 数据库中有2015年至今指数行情
2. 技术指标无明显缺失
3. 可以训练3/5/7日模型
4. 可以输出方向准确率、MAE、IC
阶段2：股吧热度

目标：

建立股民情绪和热度因子。

开发任务：

1. 采集东方财富ETF吧首页帖子
2. 写入sentiment_guba_raw
3. 按交易日聚合发帖量、评论量、阅读量
4. 计算heat_score
5. 计算heat_zscore_20d、heat_zscore_60d、heat_change_3d
6. 识别情绪阶段：冷启动、预热、发酵、高潮、分歧、兑现、退潮
7. 写入sentiment_feature_daily
8. 将舆情因子加入model_dataset_daily
9. 对比加入舆情前后的回测效果

验收标准：

1. 能每天采集股吧数据
2. 能生成热度时间序列
3. 能识别热度异常
4. 加入模型后可比较效果是否提升
阶段3：新闻采集与事件抽取

目标：

构建大事件和预期差因子。

开发任务：

1. 采集东方财富、财联社、证券时报等新闻标题
2. 写入news_raw
3. 调用大模型抽取事件结构
4. 判断event_type、event_stage、expectation_level、impact_direction
5. 计算event_score
6. 写入event_daily
7. 加入模型回测

验收标准：

1. 每日新闻能入库
2. 大模型输出稳定JSON
3. 事件可量化为event_score
4. 事件日附近预测结果可单独评估
阶段4：历史相似事件

目标：

对当前事件找到历史类似情形，统计后续3/5/7日表现。

5. 取Top 20相似事件

热度分公式：

heat_score = log(1 + post_count) + 0.5 * log(1 + comment_count) + 0.2 * log(1 + read_count)
3. 情绪阶段规则
冷启动：热度低于20日均值
预热：热度上升但未超过1倍标准差
发酵：heat_zscore_20d > 1
高潮：heat_zscore_20d > 2 且评论量快速上升
分歧：热度高但正负情绪接近
兑现：官方消息落地后，热度高且价格不涨
退潮：热度下降且价格下跌
4. 顶部风险规则
1. 热度极高 + 指数涨不动
2. 热度极高 + 成交额极高 + 官方消息落地
3. 一致看多 + 近5日涨幅较大
4. 新闻明显利好但指数冲高回落
5. 发帖量创新高但价格不创新高
5. 事件预期差分数
利多 = +1
利空 = -1
中性 = 0

预期不足 = +0.5
中性 = 0
预期充分 = -0.3
极度过热 = -0.8

超预期 = +0.8
符合预期 = 0
不及预期 = -0.8
落地兑现 = -0.6

事件分数：

event_score = direction_score * impact_strength * expectation_adjustment * stage_adjustment
九、模型设计
1. 标签
future_ret_3d = close.shift(-3) / close - 1
future_ret_5d = close.shift(-5) / close - 1
future_ret_7d = close.shift(-7) / close - 1

方向标签：

上涨：future_return > 0.008
下跌：future_return < -0.008
震荡：-0.008 <= future_return <= 0.008

后续可以改成动态阈值：

threshold = 0.4 * volatility_20d * sqrt(horizon)
2. 模型

每个周期单独训练：

LightGBMClassifier_3d
LightGBMRegressor_3d
LightGBMClassifier_5d
LightGBMRegressor_5d
LightGBMClassifier_7d
LightGBMRegressor_7d

分类模型输出方向。

回归模型输出涨跌幅。

3. 融合逻辑

最终预测由四部分组成：

技术面预测
资金/风格预测
舆情热度预测
事件/相似历史预测

初始权重：

技术面：35%
资金风格：20%
舆情热度：20%
事件相似：25%

后续根据回测结果动态调整。

十三、第一版开发重点

第一版不要做太复杂，优先完成：

1. MySQL建库建表
2. AKShare指数行情采集
3. 技术指标计算
4. 3/5/7日标签构造
5. LightGBM基础模型
6. 东方财富股吧热度采集
7. 舆情热度因子
8. 新闻采集
9. 大模型事件抽取
10. 每日预测日报

暂不做：

微博
微信公众号
抖音/B站
百度指数
分钟级高频数据
复杂交易策略
自动下单
十四、Codex开发要求

开发时请遵守：

1. 每个模块独立脚本，便于单独调试。
2. 所有数据库连接写在config/db.yaml。
3. 所有指数、ETF、期货代码写在config/symbols.yaml。
4. 所有SQL建表语句写在sql/create_tables.sql。
5. 所有日志写入logs目录。
6. 采集脚本必须支持重复运行，避免重复入库。
7. 每个数据表必须有唯一键。
8. 所有采集失败要记录日志，不要中断整个流程。
9. 大模型输出必须做JSON校验。
10. 所有模型训练结果保存到models目录。
11. 每日预测结果写入prediction_result表。
12. 每日报告输出到data/reports目录。
十五、最终验收标准

第一版完成后，应能做到：

1. 一键初始化数据库。
2. 一键采集历史行情。
3. 一键计算技术指标。
4. 一键训练3/5/7日预测模型。
5. 一键运行回测。
6. 一键采集股吧热度。
7. 一键采集新闻。
8. 一键调用大模型抽取事件。
9. 一键生成每日预测结果。
10. 一键生成Excel/HTML日报。

第一版成功标准：

1. 技术面模型可以稳定输出预测。
2. 股吧热度可以形成连续时间序列。
3. 新闻事件可以结构化入库。
4. 回测结果可以比较“只用技术指标”和“加入舆情事件”后的效果差异。
5. 每日预测报告能解释为什么看涨、看跌或震荡。
